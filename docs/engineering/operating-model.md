# 成熟团队研发制度

## 分支模型

- `main`: 生产基线，只接收已验证变更
- `develop`: 日常集成分支
- `release/*`: 版本冻结与发布候选
- `feature/*`: 需求开发分支
- `hotfix/*`: 线上修复分支

## 变更准入

- 所有变更必须通过 Pull Request
- 至少 1 位代码所有者审批
- 必须通过 CI
- 高风险变更必须带回滚方案
- 涉及接口、数据库、观测项的变更必须同步文档

## 研发节奏

1. 需求澄清：明确接口、数据、容量、回滚边界
2. 技术设计：输出变更点、风险点、兼容策略
3. 开发实现：小步提交，PR 内聚
4. 自动验证：单测、集成、烟测
5. 部署验收：先 `dev`，后 `test`，最后 `prod`
6. 发布复盘：失败与告警要沉淀进 runbook

## 非功能性门槛

- 所有服务必须暴露健康检查与 Prometheus 指标
- 所有外部请求必须带 requestId
- 所有异常必须可追溯到服务、实例、时间、接口
- 所有发布必须可回滚
- 所有环境必须可重建
